# Как сохранить пользователей при обновлении репозитория

## Проблема

По умолчанию база данных SQLite (`messenger.db`) хранится внутри Docker-контейнера. При пересоздании контейнера или обновлении репозитория все данные (пользователи, сообщения, чаты) теряются.

## Решение

Мы реализовали **персистентное хранение данных** через Docker volumes. Вот что было сделано:

### 1. Изменения в `server.py`

Добавлена поддержка переменной окружения для пути к базе данных:

```python
DB_PATH = os.environ.get("MESSENGER_DB_PATH", "messenger.db")
```

Теперь путь к базе данных можно настроить через переменную окружения.

### 2. Изменения в `Dockerfile`

- Добавлена переменная окружения `MESSENGER_DB_PATH=/app/data/messenger.db`
- Создана директория `/app/data` для хранения данных
- Директория с данными принадлежит пользователю `appuser` для безопасности

### 3. Изменения в `docker-compose.yml`

- Добавлен volume `messenger-data:/app/data` для постоянного хранения базы данных
- Добавлена переменная окружения `MESSENGER_DB_PATH`

### 4. Миграции базы данных

Создана папка `migrations/` с SQL-скриптами схемы базы данных. Это позволяет:
- Воспроизводимо создавать структуру БД
- Легко обновлять схему при изменениях

## Как это работает

```
┌─────────────────────────────────────┐
│     Docker Container                │
│  ┌───────────────────────────────┐  │
│  │  /app/data/messenger.db       │  │
│  │  (база данных)                │  │
│  └───────────────┬───────────────┘  │
│                  │                  │
│                  ▼                  │
│  ┌───────────────────────────────┐  │
│  │  Volume: messenger-data       │  │
│  │  (хранится на хосте)          │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

При перезапуске или пересоздании контейнера:
1. Docker монтирует volume `messenger-data` в `/app/data`
2. Приложение использует базу данных из этого тома
3. Данные сохраняются независимо от жизненного цикла контейнера

## Использование

### Запуск с сохранением данных

```bash
docker-compose up -d
```

### Обновление репозитория без потери данных

```bash
# Остановить текущий контейнер
docker-compose down

# Обновить код
git pull

# Пересобрать и запустить контейнер
docker-compose up -d --build
```

**Данные сохранятся**, так как volume `messenger-data` не удаляется при `docker-compose down`.

### Проверка сохранённых данных

```bash
# Посмотреть содержимое volume
docker volume inspect secure-messenger_messenger-data

# Или подключиться к базе данных
docker exec -it secure-messenger sqlite3 /app/data/messenger.db "SELECT username FROM users;"
```

### Резервное копирование

```bash
# Создать резервную копию базы данных
docker cp secure-messenger:/app/data/messenger.db backup_$(date +%Y%m%d).db

# Восстановить из резервной копии
docker cp backup_20240101.db secure-messenger:/app/data/messenger.db
```

## Локальный запуск (без Docker)

При запуске напрямую через Python база данных будет создаваться в текущей директории:

```bash
export MESSENGER_DB_PATH=./data/messenger.db
mkdir -p ./data
python server.py
```

## Переменные окружения

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| `MESSENGER_DB_PATH` | Путь к файлу базы данных | `messenger.db` |
| `PORT` | Порт сервера | `8765` |
| `HOST_PORT` | Порт на хосте | `8765` |

## Безопасность

- Volume с данными хранится отдельно от контейнера
- Данные не теряются при обновлении образа
- База данных доступна только внутри контейнера
- Для продакшена рекомендуется настроить шифрование данных на уровне хоста
